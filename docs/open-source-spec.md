# Serialport API Open Source Specification

## Purpose

`serialport-api` is a lightweight Rust service for JSON-based serial-port communication with microcontrollers and Raspberry Pi deployments.

The project is intended to replace the older JavaScript implementation in `sg-mcu-com` while preserving the useful developer workflow:

- list available serial ports
- connect/disconnect to a selected device
- send JSON commands over serial
- receive JSON or text responses from serial
- stream logs and notifications to clients
- run reliably on Raspberry Pi and embedded Linux

## Background / Migration Source

The reference JavaScript system is `avepha/sg-mcu-com`, described as вЂњJson based serial port communicationвЂќ. Its current behavior includes:

- Express HTTP API on port `4002`
- Socket.IO real-time events
- serialport connection management
- `@serialport/parser-readline` with `\r\n` delimiter
- JSON validation before commit
- mock serial device support via `/dev/ROBOT`
- saved presets in SQLite through Sequelize
- React client support through `mcu-com-client`

The current Rust repository is only a prototype. It opens one hard-coded serial port and prints UTF-8 lines. This spec defines the target production-grade open-source project.

## Goals

### Primary Goals

1. Provide a universal JSON-based serial communication service written in Rust.
2. Be simple to run on Raspberry Pi and other Linux systems.
3. Support multiple serial ports cleanly.
4. Expose a stable HTTP API for control and command submission.
5. Expose a real-time event stream for logs, responses, and raw serial data.
6. Provide excellent open-source documentation, examples, tests, and CI.

### Non-Goals for v1

- Full hardware-specific robot control logic.
- Complex GUI built into this repository.
- Cloud-hosted multi-user authentication.
- Binary/protobuf protocol support.
- GraphQL as the first API. It can be added after a stable REST/Event API exists.

## Serial Protocol

### Framing

Default line framing:

```text
JSON + \r\n
```

The service should support configurable delimiters:

- `\r\n` default for compatibility with `sg-mcu-com`
- `\n`
- custom byte delimiter in config

### Outbound Command Shape

Commands sent to the microcontroller should be JSON objects.

Minimum shape:

```json
{
  "reqId": "1",
  "method": "query",
  "topic": "sensor.read",
  "data": {}
}
```

Fields:

- `reqId`: generated by server unless client provides one explicitly
- `method`: `query`, `mutation`, `log`, `notification`, or custom method string
- `topic`: logical command topic
- `data`: command payload, any valid JSON value

### Inbound Message Shape

Microcontrollers may send:

1. JSON response with matching `reqId`
2. JSON log message
3. JSON notification message
4. Plain text line
5. Invalid JSON line

Examples:

```json
{
  "reqId": "1",
  "ok": true,
  "data": {"temperature": 28.5}
}
```

```json
{
  "method": "log",
  "data": {
    "topic": "motor",
    "level": "info",
    "message": "motor started"
  }
}
```

```json
{
  "method": "notification",
  "data": [
    {"id": "button-a", "type": "pressed", "data": {}}
  ]
}
```

## HTTP API v1

Base path: `/api/v1`

### Health

`GET /health`

Response:

```json
{"status":"ok","version":"0.1.0"}
```

### List Ports

`GET /ports`

Response:

```json
{
  "ports": [
    {
      "name": "/dev/ttyUSB0",
      "type": "usb",
      "manufacturer": "FTDI",
      "serial_number": "ABC123"
    }
  ]
}
```

### Connect Port

`POST /connections`

Request:

```json
{
  "name": "default",
  "port": "/dev/ttyUSB0",
  "baudRate": 115200,
  "delimiter": "\r\n"
}
```

Response:

```json
{
  "status": "connected",
  "connection": {
    "name": "default",
    "port": "/dev/ttyUSB0",
    "baudRate": 115200
  }
}
```

### Connection Info

`GET /connections`

Response:

```json
{
  "connections": [
    {
      "name": "default",
      "status": "connected",
      "port": "/dev/ttyUSB0",
      "baudRate": 115200
    }
  ]
}
```

### Disconnect Port

`DELETE /connections/{name}`

Response:

```json
{"status":"disconnected","name":"default"}
```

### Send Command

`POST /connections/{name}/commands`

Request:

```json
{
  "payload": {
    "method": "query",
    "topic": "sensor.read",
    "data": {}
  },
  "waitForResponse": true,
  "timeoutMs": 2000
}
```

Response when waiting:

```json
{
  "status": "ok",
  "reqId": "42",
  "response": {
    "reqId": "42",
    "ok": true,
    "data": {}
  }
}
```

Response when fire-and-forget:

```json
{"status":"queued","reqId":"42"}
```

### Event Stream

Preferred simple option:

`GET /events`

Server-Sent Event types:

- `connected`
- `disconnected`
- `serial.json`
- `serial.text`
- `serial.log`
- `serial.notification`
- `serial.error`

Example event payload:

```json
{
  "connection": "default",
  "type": "serial.log",
  "timestamp": "2026-04-29T00:00:00Z",
  "data": {
    "topic": "motor",
    "level": "info",
    "message": "motor started"
  }
}
```

WebSocket can be added later for bidirectional browser clients.

## CLI

Binary name should be short and clear. Suggested: `serialport-api`.

Commands:

```bash
serialport-api serve --host 0.0.0.0 --port 4002
serialport-api ports
serialport-api connect --port /dev/ttyUSB0 --baud-rate 115200
serialport-api send --port /dev/ttyUSB0 '{"method":"query","topic":"ping"}'
```

Environment variables:

```text
SERIALPORT_API_HOST=127.0.0.1
SERIALPORT_API_PORT=4002
SERIALPORT_API_DEFAULT_PORT=/dev/ttyUSB0
SERIALPORT_API_DEFAULT_BAUD_RATE=115200
SERIALPORT_API_DELIMITER=\r\n
```

## Configuration File

Support optional config file at:

```text
serialport-api.toml
~/.config/serialport-api/config.toml
/etc/serialport-api/config.toml
```

Example:

```toml
[server]
host = "127.0.0.1"
port = 4002
cors_origins = ["http://localhost:3000"]

[[serial.connections]]
name = "default"
port = "/dev/ttyUSB0"
baud_rate = 115200
delimiter = "\r\n"
auto_connect = false

[storage]
type = "sqlite"
path = "./serialport-api.db"
```

## Presets / Saved Commands

The JavaScript version supports saved JSON commands. Rust v1 should support this as optional local storage.

Endpoints:

- `GET /api/v1/presets`
- `POST /api/v1/presets`
- `DELETE /api/v1/presets/{id}`

Preset shape:

```json
{
  "id": 1,
  "name": "Read IMU",
  "payload": {
    "method": "query",
    "topic": "imu.read",
    "data": {}
  }
}
```

## Mock Mode

For open-source quality, the project must be testable without hardware.

Requirements:

- mock serial transport implementation
- echo mode
- scripted responses
- integration tests using mock transport

Example:

```bash
serialport-api serve --mock --mock-port /dev/ROBOT
```

## Error Model

All API errors should use consistent JSON:

```json
{
  "error": {
    "code": "serial_port_not_found",
    "message": "Serial port /dev/ttyUSB0 was not found",
    "details": {}
  }
}
```

Common codes:

- `invalid_request`
- `serial_port_not_found`
- `serial_port_open_failed`
- `serial_port_disconnected`
- `connection_not_found`
- `command_timeout`
- `invalid_json`
- `storage_error`

## Current Gap Analysis

Current Rust repository state:

- only one source file: `src/main.rs`
- hard-coded port: `/dev/cu.usbserial-DN05IJFB`
- hard-coded baud rate: `345600`
- no HTTP API implemented yet
- `rocket = "0.4.5"` is present but unused
- build currently fails on stable Rust 1.75 because Rocket 0.4 pulls `pear_codegen`, which requires nightly/dev Rust
- no tests
- no CI
- no structured error handling
- no config
- no mock mode
- no open-source metadata in `Cargo.toml`

## Recommended v1 Milestones

### Milestone 0: Repository Cleanup

- Remove unused Rocket 0.4 dependency or replace with Axum.
- Update to Rust 2021 edition.
- Add package metadata.
- Add `rustfmt`, `clippy`, and CI.
- Add initial architecture modules.

### Milestone 1: Core Protocol Library

- Implement JSON line framing.
- Parse inbound lines into `Json`, `Text`, `Log`, `Notification`, and `Error` events.
- Generate and track `reqId` values.
- Unit-test parser and framing behavior.

### Milestone 2: Serial Manager

- List ports.
- Connect and disconnect one named connection.
- Read and write framed messages.
- Add mock transport for tests.

### Milestone 3: REST API

- Add Axum server.
- Implement health, ports, connections, commands.
- Add API tests.

### Milestone 4: Event Streaming

- Add SSE endpoint.
- Broadcast JSON/text/log/notification events.
- Preserve compatibility with the event concepts from the old Socket.IO implementation.

### Milestone 5: Presets and Raspberry Pi Polish

- Add optional SQLite preset storage.
- Add Raspberry Pi install guide.
- Add systemd service example.
- Add release binaries.

## Open Questions

1. Should v1 keep exact port `4002` compatibility with `sg-mcu-com`?
2. Should the first browser-compatible stream be SSE or WebSocket?
3. Should saved presets be part of the core binary, or a later optional feature?
4. Do we want compatibility endpoints matching the JavaScript API exactly, such as `/list`, `/connect`, `/disconnect`, `/commit`, and `/info`?
5. Should this project focus only on backend service, or also include a minimal web UI later?
